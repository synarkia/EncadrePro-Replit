## The 4 findings to fix

### FINDING #3 — VR TN formula missing `majo_epaisseur` factor

**Evidence from QA:** A 10mm verre clair at 56.5 × 71cm in TN regime produced **50.90 €** but should produce **86.53 €**. The ratio 86.53 / 50.90 = exactly 1.7, which is the `majo_epaisseur` value for that product. The formula is dropping that factor.

**Expected formula for VR TN (verify and fix wherever VR TN math is computed):**

```
surface_m2 = max((largeur_cm × hauteur_cm) / 10000, mini_fact_tn)
pvuht = pauht × majo_epaisseur × coef_marge × surface_m2 × quantite
```

**Fix steps:**

1. Inspect `lib/compute-line.ts` — confirm the TN branch includes the `majo_epaisseur` multiplier. If it does (Task 03b claimed it did), then the bug is elsewhere — possibly the server-side route is using a different formula, or the frontend is reading from a computed field that bypasses `compute-line.ts`. Trace the actual number path from UI input to persisted price.

2. Inspect `artifacts/api-server/src/routes/devis.ts` — find the handler that saves/computes line prices. Task 03b explicitly flagged: *"mirroring the legacy verre formula in the server-side PUT /devis/:id/lignes so saved quote totals match the live preview"* as an outstanding follow-up. **This finding is likely a symptom of that.** If the server is recomputing prices with its own formula, replace that recomputation with a direct call to `computeLigneTotalHT()` (or equivalent) from `@workspace/compute-line` so there's ONE formula, not two.

3. **Critical**: after the fix, there should be exactly ONE place in the codebase that computes `pvuht` for a devis line. If you find two, consolidate them. This is the single most important architectural outcome of this finding — not the formula fix itself.

4. Verification: after the fix, create a devis with a verre clair 10mm line at 56.5×71, TN regime, quantité 1. Save. Reload. Query the DB directly. The stored `pvuht` should equal **86.53 €** (rounding to 2 decimal places acceptable).

**Out of scope for this finding:** the TA formula discrepancy (Finding #6) — do NOT touch the TA branch. It has its own issues that need legacy-source reconciliation.

### FINDING #1 — Persist TN/TA regime on devis lines

**Evidence from QA:** The UI exposes a TN/TA toggle on VR lines, the live preview respects it, but after save + reload the regime is gone. No column exists on `lignes_devis` to store it.

**Fix steps:**

1. Add a column to `lignes_devis` schema: `regime_pricing text` (nullable; only populated for VR lines where it's meaningful). No default; leave NULL for non-VR lines.
2. Generate migration, review SQL before applying, apply cleanly.
3. Update OpenAPI spec: add `regime_pricing: { type: string, enum: [TN, TA], nullable: true }` to the `LigneDevis` and `SaveLigneDevisBody` schemas.
4. Regenerate codegen.
5. Update the server-side save handler to accept and persist `regime_pricing`.
6. Update the UI to include `regime_pricing` in the save payload and restore it into the form state when loading a devis.
7. Verification: create a VR line, switch to TA, save, reload page. The toggle should still be on TA. The displayed price should still reflect TA regime.

**Out of scope:** the TA formula's *correctness* — Finding #6. This finding is about persistence only.

### FINDING #4 — Façonnage sub-items ignore length input

**Evidence from QA:** Façonnage lines under a main matière line are priced as `qty × unit price`, ignoring any length the user enters. The UI either has no length input or has one that's disconnected from the calculation.

**Fix steps:**

1. Inspect `lib/db/src/schema/` for the façonnage sub-items table (likely `lignes_devis_faconnage` based on earlier agent reports). If it has a length column, confirm its name. If not, add one: `longueur_cm integer` (nullable since some façonnages might be sold per-unit rather than per-length — but for FA type products with `pricing_mode === 'linear_meter'`, it should be populated).
2. Generate migration if needed.
3. Update OpenAPI spec for the façonnage sub-item schema.
4. In `QuoteLineCard.tsx` (or wherever façonnage sub-items are rendered), ensure there's a length input visible when the selected FA product has `pricing_mode === 'linear_meter'`. Default value: inherit from the parent matière line's length if that makes sense (e.g. Bords polis on a glass piece — the polished edge is naturally the perimeter of the glass). If inheritance is non-trivial, just leave the input empty and let the user type.
5. Wire the length input to the pricing formula:
   ```
   pvuht = pauht × coef_marge × (longueur_cm / 100) × quantite
   ```
6. Make sure both UI live preview AND server-side save use the same formula (same consolidation principle as Finding #3).
7. Verification: create a devis with a main matière line (Cadre Trafalia, Ref 420) + a façonnage sub-item (Bords polis, Ref 7) with longueur 280cm, quantité 1. Expected façonnage line total: `13.50 × 1.0 × 2.80 × 1 = 37.80 €`. Save, reload, confirm persisted.

### FINDING #5 — Catalogue product cards aren't clickable

**Evidence from QA:** On `/catalogue`, clicking a product does nothing. No link to detail or edit. Thomas cannot review or modify his 5,000 products without this.

**Fix steps:**

1. In `artifacts/encadrepro/src/pages/catalogue/index.tsx`, wrap each product card in a link to `/catalogue/[id]` (adjust to actual routing convention — check existing route patterns in the project).
2. Check if `pages/catalogue/[id].tsx` already exists. If not, create a minimal version:
   - Shows all product fields in a read-mostly layout
   - Groups fields into sections: "Identification" (ref_legacy, designation, type_code, fournisseur), "Pricing" (pauht, coef_marge, pricing_mode + type-specific pricing fields), "Type-specific" (the VR/FA/EN fields), "Metadata" (notes, image_url, vendu, created_at, updated_at)
   - A "Modifier" button that opens an edit dialog (reuse the QuickAddProductModal component in edit mode — if QuickAdd supports an optional `initialValues` prop, use it; if not, extract the shared form into a separate component that both QuickAdd and this edit page can use)
   - A "Retour au catalogue" breadcrumb/back button
3. Match existing page layout patterns — don't invent new styling.
4. Verification: navigate to `/catalogue`, click a VR product — should navigate to `/catalogue/[id]`, show all VR-specific fields (épaisseur, majo, mini fact TN/TA, coef marge TA, plus-value TA%). Click Modifier — edit form opens with current values pre-filled. Save a change — values persist.

**Out of scope:** delete functionality, product duplication, image upload, advanced filtering on the list page.
